PART A — No-Canvas Fallback Strategy (Client-Only, Max Reliability)
Goal: Maximise Safari trust without canvas, while staying inside the “gold handler” constraints.
This does not promise ShareLah parity — it raises the success rate and removes self-inflicted failures.

A1. Core Principle (Non-Negotiable)
If the source image is already a valid JPEG/PNG under 800KB, never touch it.
Canvas is a mutation. Mutation = risk on iOS Safari.

A2. Image Trust Ladder (Decision Tree)
Step 1 — Fetch & Inspect (no mutation)
async function preloadShareImageNoCanvas(url) {
  const res = await fetch(url, { cache: 'force-cache' });
  const blob = await res.blob();

  const isSupportedType =
    blob.type === 'image/jpeg' || blob.type === 'image/png';

  const isSizeOk = blob.size < 800 * 1024;

  if (!isSupportedType || !isSizeOk) {
    console.warn('[PRELOAD] unsupported blob', blob.type, blob.size);
    return null;
  }

  const file = new File([blob], 'share.jpg', {
    type: blob.type,
    lastModified: Date.now(),
  });

  const canShare =
    navigator.canShare &&
    navigator.canShare({ files: [file] });

  console.log('[PRELOAD NO CANVAS] canShare =', canShare);

  return canShare ? file : null;
}
Why this works better
* No canvas
* No re-encoding
* Original binary stays intact
* Matches how Safari expects camera/library images

Step 2 — Canvas ONLY as last resort
Use canvas only if:
* PNG > 800KB
* JPEG > 800KB
* Unsupported format (e.g. webp)
If canvas path fails → hard stop (no silent downgrade).
