🚨 The critical mistake (this is the missing piece)
❌ Safari iOS does NOT reliably accept a raw Blob for image clipboard writes
Even if:
* Blob is preloaded
* Blob is JPEG
* Blob size is small
* Gesture is correct
👉 On iOS Safari, images must be supplied as a Promise<Blob> inside ClipboardItem.
This is the unintuitive part.

Why this matters (and why ShareLah works)
Safari internally expects clipboard images to be asynchronously materialized, even if they’re already in memory.
If you pass:
'image/jpeg': blob
Safari may reject with:
复制失败 / 请重试
If you pass:
'image/jpeg': Promise.resolve(blob)
Safari accepts it.
This is a Safari implementation quirk, not spec logic.
📌 ShareLah almost certainly still uses a Promise — just one that resolves immediately.

✅ Correct ClipboardItem construction (THIS is the fix)
Change only this part:
❌ Current (causes failure)
const clipboardItem = new ClipboardItem({
  'image/jpeg': preloadedImageBlob,
  'text/plain': textBlob,
});
✅ Correct for iOS Safari
const clipboardItem = new ClipboardItem({
  'image/jpeg': Promise.resolve(preloadedImageBlob),
  'text/plain': textBlob,
});
That’s it.
No fetch No await No async handler No timing change
But Promise-wrapped Blob.

Why this does NOT break gesture context
Important clarification:
* The call to clipboard.write() is still synchronous
* Safari allows Promises inside ClipboardItem
* What Safari rejects is:
    * Promises that depend on network
    * Or clipboard.write called after await
You are doing neither.
So this is safe.
