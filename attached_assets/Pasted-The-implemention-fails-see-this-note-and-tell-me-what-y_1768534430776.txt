1️⃣ Clipboard MIME + Promise
Even small differences break XHS behavior:
* ShareLah always ensures:
    * JPEG only
    * No await, always Promise-based
    * Single ClipboardItem
    * MIME key matches blob exactly
* Check your blob type: fetch(proxyUrl).then(r => r.blob()).then(b => console.log(b.type));
* 
    * If it’s image/jpeg, good
    * If image/png or image/octet-stream, XHS ignores it → opens picker
Tip: ShareLah may force-convert PNG → JPEG server-side before sending to client.

2️⃣ Clipboard write inside real click handler
* ShareLah does:
button.addEventListener('click', () => {
    // all clipboard + deep link happens directly here
});
* No await before creating ClipboardItem, no setTimeout >0 before navigator.clipboard.write, no async wrapper
* If your code uses:
    * async function share() { ... await ... }
    * or setTimeout(() => navigator.clipboard.write(...), 0) too early → iOS loses the “user gesture context” and XHS ignores the image
Check: your function must be entirely synchronous inside click except for Promise-based blob fetch.

3️⃣ Image Proxy timing / CORS / headers
* ShareLah ensures the image is fast (<100ms fetch ideally)
* Headers are exact:
    * Content-Type: image/jpeg
    * Cache-Control: no-store
    * Content-Disposition: inline; filename="share.jpg"
* iOS Safari will ignore images if fetch is slow or headers incorrect
* Even Access-Control-Allow-Origin can interfere if your server isn’t set correctly
Check: Open proxy URL in Safari → long-press → copy image → paste in Notes
* If it works → proxy OK
* If not → proxy headers / timing problem

4️⃣ Deep link
* ShareLah uses exactly:
xhsdiscover://post_note?ignore_draft=true
* Confirm your link is exactly the same, no extra params, no redirect
* Any difference (even xhsdiscover://post_note?draft=false) → XHS may fall back to picker

5️⃣ Hidden ShareLah tricks
From observations and tests:
1. Backend ensures small, fast JPEG (200–800 KB)
    * Large images sometimes get ignored
2. ClipboardItem Promise resolves quickly
    * Even ~200ms delay between Promise creation and clipboard.write may cause fallback
3. One ClipboardItem only
    * Multiple items or multiple writes sometimes triggers picker fallback
4. Deep link timing
    * ShareLah triggers deep link after micro-delay (~20ms) to allow Safari to commit clipboard
